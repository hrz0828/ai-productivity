Cursor 是一个集成 AI 能力的代码编辑器，适合在熟悉的 IDE 环境中完成项目问答、代码生成、重构和辅助调试。对于产品型开发者和独立开发者来说，Cursor 的价值不只是“帮你写代码”，更重要的是帮助你理解项目上下文，并在明确范围内小步修改。

新手使用 Cursor 最常见的问题，是一上来就让它“重构整个项目”或“帮我做一个完整功能”。更稳妥的方式是：先问清项目结构，再限定修改范围，最后检查 diff 和运行验证。

## 工具简介

Cursor 是一个集成 AI 能力的代码编辑器，适合在熟悉的编辑器环境中完成代码问答、生成、修改和重构。

适合：

- 项目问答和代码理解
- 生成组件、函数和测试
- 小范围重构
- 辅助调试报错
- 为现有代码补充注释或文档
- 学习陌生技术栈

不适合：

- 不看 diff 就直接接受所有修改
- 一次性让 AI 大改核心架构
- 没有测试就改支付、权限、数据迁移等高风险逻辑
- 把 AI 的解释当成真实业务规则

## 适合谁使用

- 希望提高编码效率的开发者
- 产品型创业者
- 需要维护小型项目的独立开发者
- 正在学习编程的人
- 需要在现有代码库中快速定位功能的人

如果你是非专业开发者，Cursor 可以帮助你进入项目，但仍然需要学会看 diff、运行测试和理解错误信息。

## 常见使用场景

### 项目问答

你可以询问某个功能在哪里实现、某个文件的作用、调用链是什么。

示例：

```text
请解释这个项目中内容文章是如何从 Markdown 渲染到页面的。先不要修改代码。
```

### 代码生成

适合生成组件、工具函数、测试用例和重复性样板代码。

关键是提供上下文和约束：使用什么框架、现有风格、输入输出、是否需要测试。

### 重构

可以让 Cursor 解释重构风险，再分步骤修改，避免一次性大改。

### 调试

把报错信息、相关代码和你已经尝试过的步骤提供给 Cursor，让它先定位原因，再给最小修复建议。

## 推荐使用流程

### Step 1：用项目问答理解上下文

先问 Cursor 某个功能在哪里、相关文件有哪些、调用链是什么。

```text
请帮我解释这个项目的文章列表页是如何读取内容并渲染卡片的。先不要修改代码。
```

这一步可以避免 Cursor 在不了解项目结构时乱改文件。

### Step 2：限定修改范围

让 Cursor 修改代码前，明确范围和约束：

```text
请只修改 ArticleCard 组件，不要改动内容 schema，也不要引入新依赖。
```

可用约束：

- 保持现有 API 不变
- 不引入新依赖
- 不修改数据库结构
- 只改指定文件
- 保持 Tailwind 风格一致
- 修改后补充测试或检查清单

### Step 3：让它解释风险

对于重构任务，先让 Cursor 解释可能影响的地方：

```text
如果要重构这个组件，可能影响哪些页面？需要检查哪些状态？请先给风险清单，不要修改代码。
```

### Step 4：小步修改

把任务拆成多个小步骤：

1. 先改组件结构
2. 再接入页面
3. 再处理空状态
4. 最后运行构建或测试

小步修改比一次性生成大 diff 更容易审查。

### Step 5：人工审查 diff

Cursor 生成代码后，一定要检查 diff，特别是：

- 删除文件
- 重命名路径
- 新增依赖
- 改动配置
- 修改权限、支付、登录、数据逻辑
- 隐式改变现有行为

## 使用案例：用 Cursor 生成一个文章卡片组件

### 场景

你正在开发内容站，需要一个 ArticleCard 组件展示标题、描述、分类、标签和发布时间。

### 示例指令

```text
请根据当前项目的 Tailwind 风格，生成一个 Astro ArticleCard 组件。

要求：
- 展示 title、description、category、tags 和日期
- 适配移动端
- 长标题要换行
- 不引入新依赖
- 保持现有链接路径规则
```

### 输出结果

Cursor 可以根据当前文件上下文生成组件代码，并帮助你把它接入列表页。

### 人工检查

生成组件后，要检查：

- props 类型是否正确
- 链接路径是否正确
- 移动端布局是否可读
- 标签数量很多时是否换行
- 没有 description 时是否有兜底
- 是否符合项目已有样式

## 使用案例：让 Cursor 调试构建错误

```text
下面是 npm run build 的错误日志。请帮我定位原因。

要求：
- 先指出最关键的错误行
- 判断是类型、路径、依赖还是内容数据问题
- 给出最小修复建议
- 不要建议大规模重构

错误日志：
[粘贴]
```

如果错误来自内容文件，Cursor 可能需要查看 frontmatter、schema 或路由生成逻辑。不要只给最后一行错误。

## 常见误区

- 让 Cursor 一次性重构太多文件。
- 没有说明不要引入新依赖。
- 只看代码能运行，不检查边界状态。
- 没有检查生成代码是否符合项目风格。
- 把 AI 的解释当作真实业务规则。
- 没有运行测试或构建。
- 对删除、部署、数据库迁移等操作没有额外确认。

## 常见失败案例

### 失败 1：改动范围失控

你只想改一个按钮样式，Cursor 顺手改了组件结构、主题变量和多个页面。这时应该撤回，重新要求：

```text
请只修改这个按钮的 class，不要改组件结构和其他文件。
```

### 失败 2：生成代码不符合项目风格

如果项目使用 Astro + Tailwind，Cursor 生成了 React 组件或 CSS Modules，说明上下文或约束不够。需要明确技术栈和现有文件风格。

### 失败 3：没有处理空状态

列表、搜索、表单等功能经常漏掉空状态、加载状态和错误状态。可以追加：

```text
请检查这个组件是否覆盖空状态、长文本、移动端和错误数据。
```

## 人工复核清单

- 是否只修改了预期文件？
- props、类型和导入路径是否正确？
- 移动端和空状态是否正常？
- 是否破坏现有 API？
- 是否需要补测试？
- 是否有不必要的新依赖？
- 是否改变了配置、构建或部署行为？
- 是否运行了构建或相关测试？

## Cursor 和 Claude Code 怎么搭配

一种常见搭配方式：

- Cursor：适合在编辑器里做局部修改、查看上下文、生成组件和调试。
- Claude Code：适合终端内跨文件任务、运行命令、批量修改、项目级计划和验证。

如果任务只涉及一个组件，Cursor 很方便；如果任务需要搜索多个文件、修改内容集合、运行构建和总结变更，Claude Code 更适合。

## 结论

Cursor 的最佳用法不是让 AI 接管项目，而是把它当成理解代码和小步修改的助手。先问上下文，再限定范围，最后检查 diff 和运行验证。这样既能提高效率，也能降低 AI 误改代码的风险。
