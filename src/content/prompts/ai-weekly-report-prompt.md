写周报看起来是一个简单任务，但很多人真正卡住的地方并不是“不会写”，而是本周信息太碎：会议记录、任务进展、临时支持、数据结果、风险问题分散在不同工具里。直接把这些内容丢给 AI，往往会得到一份语气很顺但不够真实的周报：成果被夸大、风险被弱化、下周计划过于空泛。

这个 Prompt 的目标不是让 AI 替你编故事，而是让它把你已经做过的事情整理成结构清晰、可复核、适合汇报的文字。使用前最好先准备原始工作记录，使用后再人工检查事实、数据和风险表达。

## 适用场景

当你需要快速整理本周工作内容、输出结构清晰的周报，或者把零散事项变成可以汇报的文字时，可以使用这个 Prompt。

比较适合的场景包括：

- 每周固定向直属领导发送周报
- 项目组需要同步本周进展和风险
- 运营、产品、研发、设计等角色需要总结阶段性成果
- 工作记录很多，但没有时间重新组织语言
- 希望从“做了什么”升级为“产生了什么结果”

不太适合的场景包括：

- 本周没有任何明确工作记录，只想让 AI 编一份看起来完整的周报
- 周报涉及敏感数据、客户隐私或尚未公开的商业信息
- 需要精确引用财务、法务、合同或绩效数据，但你没有核对来源
- 公司已经有强制格式，且字段与本文 Prompt 完全不同

## 适合人群

- 产品经理
- 运营人员
- 项目经理
- 研发和设计同学
- 需要定期写周报的知识工作者

如果你是管理者，也可以把这个 Prompt 改成“团队周报汇总 Prompt”，让 AI 先按成员、项目、风险和待协调事项归类，再由你确认重点。

## 使用前先准备什么

不要一上来只输入“帮我写周报”。我建议先整理 4 类材料：

1. **完成事项**：本周实际完成了什么，不需要一开始就写得漂亮。
2. **结果和数据**：有没有上线、交付、转化、访问量、缺陷数、响应时长等可验证结果。
3. **问题和风险**：哪些事情没完成、为什么没完成、需要谁协助。
4. **下周计划**：最好写成具体动作，而不是“继续推进项目”。

一个可用的原始记录不需要很长，例如：

```text
- 上线用户反馈表单
- 整理 18 条反馈，分成帮助文档缺失、路径不清晰、功能疑问三类
- 帮助文档首页改版，入口更明显
- 数据埋点还没完全验证
- 开发资源紧张，反馈看板可能延期
- 下周补数据看板，继续优化反馈流程
```

这些材料越具体，AI 输出越不容易变成空话。

## Prompt 正文

```text
你是一名专业的职场写作助理。请根据我提供的本周工作信息，帮我生成一份结构清晰、表达专业、重点突出的工作周报。

请按照以下结构输出：
1. 本周核心成果
2. 重点工作进展
3. 数据或结果说明
4. 遇到的问题与风险
5. 下周计划
6. 需要协同或支持的事项

写作要求：
- 语言简洁、正式，适合发给直属领导或项目组
- 不要夸大事实
- 对零散事项进行归类和提炼
- 如果信息不足，请用“待补充”标记
- 区分已经完成、正在推进、存在风险的事项
- 不要编造没有提供的数据和结论

以下是我的原始信息：
[在这里粘贴本周工作记录]
```

## 输入示例

```text
本周完成：上线用户反馈表单，整理了 18 条反馈，优化了帮助文档首页。
遇到问题：开发资源紧张，数据埋点还没有完全验证。
数据结果：帮助文档访问量提升约 12%。
下周计划：继续优化反馈流程，补充数据看板。
```

## 输出示例

```markdown
# 本周工作周报

## 1. 本周核心成果
本周完成用户反馈表单上线，并整理首批 18 条用户反馈，为后续帮助文档和产品体验优化提供了基础输入。

## 2. 重点工作进展
- 上线用户反馈表单，完成基础提交流程验证。
- 整理用户反馈内容，初步归类为功能疑问、使用路径和文档缺失三类。
- 优化帮助文档首页，使用户更容易找到常见问题入口。

## 3. 数据或结果说明
帮助文档访问量较上周提升约 12%，后续需要结合停留时间和反馈提交率继续观察。

## 4. 遇到的问题与风险
数据埋点仍未完全验证，短期内对反馈转化率的判断可能不够准确。开发资源紧张也可能影响后续优化节奏。

## 5. 下周计划
- 继续优化反馈流程。
- 补充数据看板。
- 验证核心埋点是否准确。

## 6. 需要协同或支持的事项
需要开发同学协助确认埋点数据是否完整，并评估反馈看板排期。
```

## 常见失败输出和修正方法

### 失败 1：AI 把小成果写得过大

常见输出：

```text
本周显著提升了帮助文档体验，并建立了完整的用户反馈体系。
```

问题在于“显著”和“完整体系”都超过了原始材料。更稳妥的写法是：

```text
本周完成用户反馈表单上线，并开始整理首批反馈，为后续优化帮助文档提供输入。
```

如果经常出现夸大，可以在 Prompt 中追加：

```text
请使用克制表达，不要使用“显著、全面、完整、极大提升”等没有数据支撑的词。
```

### 失败 2：风险写得太轻

常见输出：

```text
数据埋点仍在完善中。
```

这句话没有说明影响。更好的表达是：

```text
数据埋点尚未完全验证，短期内对反馈转化率和页面优化效果的判断可能不够准确。
```

风险不是为了“显得有问题”，而是帮助接收周报的人判断是否需要协调资源。

### 失败 3：下周计划太空泛

常见输出：

```text
下周继续推进相关优化工作。
```

可以要求 AI 把下周计划改成动作列表：

```text
请把下周计划改成 3-5 个可执行动作，每个动作尽量包含对象和结果。
```

## 调试心得：如何让周报更像你自己写的

我通常会做三步调试：

1. **先让 AI 生成结构**：不用急着追求完美表达，先看信息有没有归类正确。
2. **再删掉夸张词**：尤其是“显著提升、全面优化、成功打造”这类没有数据支撑的词。
3. **最后改语气**：如果发给领导，保留结果和风险；如果发给项目组，保留更多协作细节。

你也可以补充自己的语气偏好：

```text
请使用简洁、克制、偏事实陈述的语气，不要写成宣传稿。
```

或者：

```text
请保留适度口语化，适合发在团队群，不要太正式。
```

## 可调整参数

- 周报语气：正式、简洁、口语化
- 周报长度：300 字、500 字、800 字
- 汇报对象：直属领导、项目组、跨部门团队
- 输出格式：Markdown、表格、邮件格式
- 重点方向：结果导向、风险同步、过程复盘、资源协调

## 变体 Prompt：给管理者汇总团队周报

```text
你是一名项目管理助理。请根据多位成员的本周工作记录，整理一份团队周报。

请输出：
1. 本周团队核心成果
2. 按项目归类的进展
3. 本周关键数据或交付物
4. 主要风险和阻塞
5. 下周团队重点
6. 需要管理者协调的事项

要求：
- 不按成员流水账罗列，优先按项目和目标归类
- 不编造没有提供的数据
- 对风险说明可能影响
- 标记信息不足的地方

团队原始记录：
[粘贴多位成员记录]
```

## 发送前检查清单

- 是否有夸大或未经确认的数据？
- 风险描述是否足够清楚？
- 下周计划是否具体到可执行事项？
- 是否遗漏需要领导或其他团队支持的内容？
- 是否需要删除过多细节，让周报更聚焦结果？
- 是否包含不应该在周报中公开的客户、合同、账号或个人信息？
- 是否把“正在推进”误写成了“已经完成”？

## 常见问题

### 输入内容很乱怎么办？

直接粘贴原始记录即可，但最好用换行区分事项。AI 会先归类，再生成周报。如果内容特别多，可以先让 AI 做“信息归类”，再生成正式周报。

### 周报是否可以直接发送？

不建议完全不检查就发送。至少检查事实、数据、风险和下周计划。AI 很擅长把语言写顺，但它不知道哪些内容在你的组织里属于敏感信息，也不知道某个数字是否已经被验证。

### 如果本周事情很多，周报会不会太长？

可以追加要求：

```text
请把最终周报控制在 500 字以内，保留最重要的 3 个成果和 2 个风险。
```

### 如果本周事情很少怎么办？

不要让 AI 硬编。可以把周报重点放在“当前状态、阻塞原因、下周动作”。真实、清楚，比看起来很满更重要。
