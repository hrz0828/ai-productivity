PRD 最容易被 AI 写得“像那么回事”，但真正能进入评审的 PRD 不是一份漂亮文档，而是一份能让设计、研发、测试、运营都理解边界的协作材料。AI 可以帮你搭结构、补充问题、整理功能点，但不能替你确认业务事实、用户优先级和技术约束。

这篇文章提供的 Prompt 适合把功能想法整理成 PRD 初稿。它的价值不在于一次生成终稿，而在于帮你快速暴露缺失信息：目标用户是否清楚、场景是否具体、功能边界是否能被研发理解、数据指标是否真的可衡量。

## 适用场景

当你已经有一个功能想法，但还没有完整 PRD 结构时，可以使用这个 Prompt 快速生成初稿。

适合使用的情况：

- 已经有用户反馈、业务目标或内部需求，但表达比较零散
- 想在评审前先整理出 PRD 框架
- 需要把“功能想法”转成“背景、目标、流程、规则、验收标准”
- 希望 AI 帮你列出待确认问题，避免直接带着模糊需求进入开发

不适合直接使用的情况：

- 需求是否成立还没有验证
- 只有一句“我要做一个搜索功能”，没有目标用户和业务目标
- 涉及支付、权限、安全、合规等高风险功能，但没有专业人员参与
- 希望 AI 替你决定产品策略或研发排期

如果需求还不清楚，先不要直接写 PRD。可以先使用 [用 AI 做需求分析的完整工作流](/workflows/demand-analysis-with-ai/) 判断需求是否真实、是否值得做，再进入 PRD 写作。也可以从 [AI 产品经理专题](/topics/ai-product-manager/) 按推荐顺序阅读。

## 使用前需要准备的信息

至少准备以下 6 类材料：

1. **需求来源**：用户反馈、业务目标、内部效率问题还是竞品启发。
2. **目标用户**：谁会使用这个功能，他们现在怎么解决问题。
3. **核心场景**：用户在什么时刻、为了什么目标触发这个需求。
4. **成功标准**：上线后如何判断有效，不一定是复杂指标，但要可观察。
5. **约束条件**：时间、技术、合规、数据、权限、依赖系统。
6. **不做范围**：第一版不解决什么，避免 PRD 无限膨胀。

如果这些信息都没有，AI 仍然可以生成一份 PRD，但那更像“示范文档”，不是可执行需求。

## Prompt 正文

```text
你是一名资深产品经理。请根据我提供的功能想法，生成一份产品需求文档初稿。

请包含：
1. 需求背景
2. 目标用户
3. 用户问题
4. 产品目标
5. 核心流程
6. 功能需求
7. 非功能需求
8. 数据指标
9. 风险与依赖
10. 待确认问题
11. 不做范围
12. 验收标准

要求：
- 明确区分事实、假设和待确认问题
- 不要编造业务数据
- 功能描述要可被设计和研发理解
- 对不完整的信息提出澄清问题
- 不要把第一版需求写得过大
- 涉及权限、数据、通知、删除、发布等动作时，请主动提示风险

功能想法：
[粘贴内容]
```

## 使用案例：为内容站增加搜索功能

### 场景

你运营一个 AI 内容站，希望增加站内搜索，用户可以搜索 Prompt、Workflow 和工具教程。

### 示例输入

```text
功能想法：增加站内搜索页面，用户输入关键词后可以搜索所有文章。
目标用户：访问内容站的普通读者，尤其是带着具体任务来的用户。
目标：提高内容发现效率，减少用户只看首页就离开。
约束：第一版不做登录，不记录用户搜索历史，使用静态站可部署的方案。
不做范围：不做个性化推荐，不做收藏，不做复杂筛选。
```

### 期望输出

AI 应生成 PRD 初稿，包括搜索入口、搜索结果展示、无结果状态、移动端适配、数据指标和不做范围。

### 人工复核重点

PRD 需要明确技术方案是否依赖第三方服务、搜索结果是否需要高亮、是否支持中文分词。这些问题不能只停留在“支持搜索”四个字。

## 输出示例

```markdown
# 站内搜索功能 PRD 初稿

## 1. 背景
随着内容数量增加，用户仅通过栏目浏览查找 Prompt 和 Workflow 的效率下降，需要提供站内搜索能力。

## 2. 产品目标
帮助用户通过关键词快速找到相关文章，提升内容发现效率和页面浏览深度。

## 3. 目标用户
- 首次访问本站、带着明确问题搜索的用户
- 想快速找到某个 Prompt 模板的老用户

## 4. 核心流程
1. 用户从首页或导航进入搜索页。
2. 用户输入关键词。
3. 页面展示匹配文章标题、摘要、类型和链接。
4. 用户可以点击结果进入详情页。
5. 如果没有结果，页面提供热门关键词和任务入口。

## 5. 功能需求
### 搜索入口
首页 Hero 区和 Header 提供搜索入口。

### 搜索页面
用户输入关键词后展示匹配文章标题、摘要和链接。

### 类型筛选
支持按 Prompt、Tool、Workflow、Guide 等类型筛选。

### 无结果状态
当没有结果时，提示用户更换关键词，并提供热门栏目入口。

## 6. 不做范围
- 不做用户搜索历史
- 不做个性化推荐
- 不做登录态收藏

## 7. 数据指标
- 搜索入口点击次数
- 搜索关键词提交次数
- 搜索结果点击率
- 无结果搜索次数

## 8. 待确认问题
- 是否需要中文分词优化？
- 是否统计搜索关键词？
- 是否需要结果高亮？
```

## 常见失败输出和修正方法

### 失败 1：AI 把功能写得过大

如果输入只有“增加站内搜索”，AI 可能会自动加上搜索历史、个性化推荐、热门榜单、登录收藏。这些功能看起来完整，但会让第一版需求失控。

可以追加：

```text
请严格控制第一版范围，只保留能验证核心价值的功能。把增强功能放到后续版本。
```

### 失败 2：缺少验收标准

很多 AI 生成的 PRD 会写“支持搜索文章”，但研发和测试无法判断完成标准。你可以要求：

```text
请为每个核心功能补充可测试的验收标准，使用 Given / When / Then 或列表形式。
```

例如：

```text
当用户输入“周报”时，搜索结果应展示标题或正文包含“周报”的文章，并显示标题、摘要和跳转链接。
```

### 失败 3：把假设写成事实

AI 可能会写：

```text
大量用户反馈当前内容查找效率低。
```

如果你没有真实反馈，这句话就是风险。更稳妥的表达是：

```text
随着内容数量增加，存在用户查找效率下降的风险，需通过搜索行为数据验证。
```

## 调试心得：先让 AI 问问题，再让 AI 写 PRD

我更推荐两段式用法：

第一步，让 AI 只提问：

```text
请不要直接写 PRD。请先根据我的功能想法，列出你需要确认的 10 个关键问题，并按业务目标、用户场景、功能边界、数据指标、技术依赖分类。
```

第二步，在回答关键问题后，再让 AI 生成 PRD 初稿。

这样做的好处是：AI 不会过早“补全”不存在的信息，你也能在写 PRD 前发现需求里的空洞。

## 变体 Prompt：让 AI 做 PRD 评审

```text
你是一名严格的产品评审专家。请审查下面的 PRD 初稿，找出可能影响设计、研发、测试和上线的风险。

请从以下维度检查：
1. 背景和目标是否清楚
2. 目标用户和场景是否具体
3. 功能边界是否明确
4. 是否存在未说明的权限、数据、安全或通知逻辑
5. 验收标准是否可测试
6. 是否有过度设计
7. 哪些内容属于事实、假设和待确认问题

请输出：
- 主要问题清单
- 必须补充的信息
- 建议删减的范围
- 修改后的 PRD 结构建议

PRD 初稿：
[粘贴 PRD]
```

## 人工复核清单

发布或评审 PRD 前，至少检查这些问题：

- 需求来源是否真实？
- 目标用户是否具体到角色或场景？
- 是否有明确不做范围？
- 功能规则是否能被研发拆解？
- 验收标准是否能被测试执行？
- 是否把假设写成了事实？
- 是否涉及权限、数据、删除、通知、支付、发布等高风险动作？
- 是否需要设计、研发、法务、安全或运营提前评审？

## 使用边界

AI 可以帮助你把 PRD 写得更快，但不能替你决定需求是否值得做。真正重要的判断仍然来自用户研究、业务目标、资源约束和团队共识。把 AI 生成的 PRD 当成“初稿”和“检查清单”，而不是直接进入研发的最终需求。
