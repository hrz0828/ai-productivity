## 这个 Skill 解决什么问题

PRD 写作 Skill 用于把零散需求信息整理成结构化产品需求文档初稿，减少产品经理在文档框架、表达组织和遗漏检查上的重复劳动。

它尤其适合这些情况：

- 需求讨论后，只有会议记录和零散想法，需要快速整理成 PRD 初稿。
- 功能边界还不完全清楚，需要先列出待确认问题。
- 产品经理需要把业务目标、用户场景和研发可执行需求连接起来。
- 需求评审前，需要检查是否遗漏权限、异常、数据指标和依赖。
- 需要把一个大需求拆成 MVP 版本和后续迭代版本。

## 输入要求

使用前建议准备以下信息：

- 业务背景：为什么现在要做这个需求。
- 目标用户：谁会使用，用户角色有哪些差异。
- 用户痛点：当前遇到的问题或机会是什么。
- 产品目标：希望通过功能达成什么结果。
- 核心功能：必须包含哪些能力。
- 使用场景：用户从哪里进入，完成什么任务。
- 约束条件：时间、人力、技术、合规或业务限制。
- 期望上线时间：是否有固定节点。
- 已知风险：依赖、边界、数据口径、权限等。

## 使用方法

### 第一步：先让 AI 反问缺失信息

PRD 最怕信息不完整但写得很完整。建议先让 Skill 检查输入，并列出关键缺口。

```text
请先判断这些需求信息是否足够写 PRD。如果不足，请列出必须确认的问题，不要直接编造。
```

### 第二步：生成 PRD 初稿

当背景和目标相对明确后，再让 AI 输出结构化文档。

```text
请根据已有信息生成 PRD 初稿，重点写清楚用户场景、功能需求、异常状态和数据指标。
```

### 第三步：拆分用户故事和验收标准

PRD 初稿完成后，可以让 AI 继续拆成研发更容易理解的用户故事。

```text
请把功能需求拆成用户故事，并为每个故事补充验收标准。
```

### 第四步：做评审前缺口检查

最后让 AI 从研发、设计、测试、运营视角检查遗漏项。

```text
请分别从研发、设计、测试、运营视角检查这份 PRD 还缺什么。
```

## 推荐 Prompt 模板

```text
你是一个 PRD 写作 Skill。请基于以下信息生成产品需求文档初稿。

业务背景：
目标用户：
用户痛点：
产品目标：
核心功能：
使用场景：
约束条件：
期望上线时间：
已知风险：

要求：
1. 不要编造未提供的信息，缺失内容放到“待确认问题”
2. 功能需求要写到可评审的粒度
3. 补充异常状态、权限规则、数据指标和依赖
4. 输出用户故事和验收标准
```

## 输出格式

```markdown
# 产品需求文档

## 1. 背景
## 2. 目标
## 3. 用户与场景
## 4. 用户故事
## 5. 功能需求
## 6. 非功能需求
## 7. 权限与边界
## 8. 数据指标
## 9. 风险与依赖
## 10. 验收标准
## 11. 待确认问题
```

## 使用案例一：生成 Prompt 收藏功能 PRD

### 场景

内容站已有多篇 Prompt 和 Workflow，用户希望保存常用内容。你需要快速写出收藏功能 PRD 初稿。

### 输入示例

```text
业务背景：用户需要反复访问常用 Prompt。
目标用户：经常使用本站模板的读者。
用户痛点：每次都要通过搜索或目录重新找到内容。
产品目标：提升复访效率和内容使用频率。
核心功能：收藏文章、查看收藏列表、取消收藏。
约束：第一版不做复杂分组，不做公开分享。
```

### Skill 输出方向

AI 应输出背景、目标、用户故事、功能流程、页面说明、数据指标和待确认问题。对于登录依赖、数据存储、空状态和取消收藏流程，也应明确标记。

### 评审前检查

- 是否说明收藏入口在哪里？
- 是否说明未登录用户如何处理？
- 是否有取消收藏流程？
- 是否定义成功指标？
- 是否明确第一版不做什么？

## 使用案例二：生成内容审核功能 PRD

### 场景

内容团队希望新增发布前审核流程，避免标题、SEO 描述、标签和内链遗漏。

### 输入示例

```text
业务背景：站点内容增加后，人工发布容易漏掉 SEO 字段和内链。
目标用户：内容编辑、站长。
核心功能：发布前检查 frontmatter、标题、描述、标签、内链、构建结果。
约束：第一版只做检查清单，不做复杂审批流。
```

### 使用建议

这类需求可以让 AI 输出“检查项列表 + 触发时机 + 通过标准”。如果涉及自动化检查，还要补充错误提示和失败后的处理方式。

## 常见错误

- 只写功能列表，没有说明用户场景。
- 写了很多愿景，但没有验收标准。
- 没有标记“不做什么”，导致范围膨胀。
- 忽略权限、异常状态和空状态。
- 数据指标只写“提升体验”，没有可衡量口径。

## 注意事项

AI 生成的 PRD 是初稿，不应直接替代需求评审。对于边界条件、数据口径、权限规则、异常状态和技术依赖，需要产品经理继续补充，并和研发、设计、测试一起确认。
