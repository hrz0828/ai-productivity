## 这个 Skill 解决什么问题

团队里很多工作依赖个人经验，交接时容易遗漏。SOP 生成 Skill 可以把零散经验、操作记录和团队规范转成标准流程，降低重复沟通成本，也方便新人按步骤执行。

它适合这些场景：

- 内容发布、活动上线、客户交付等重复性流程。
- 老员工经验沉淀和新人培训。
- 把一次成功项目复盘成可复用流程。
- 把“口头约定”整理成团队检查清单。
- 为自动化工作流准备清晰步骤和异常分支。

## 输入要求

- 工作目标：这套流程最终要完成什么。
- 适用范围：什么情况适用，什么情况不适用。
- 当前操作步骤：现有做法、截图说明或口述记录。
- 常见异常：哪些情况容易失败或返工。
- 质量标准：做到什么程度才算完成。
- 负责人角色：谁执行，谁审核，谁确认结果。
- 工具和材料：需要用到哪些系统、模板、账号或文件。

## 使用方法

### 第一步：输入真实操作过程

不要只输入“帮我写内容发布 SOP”。最好把真实操作步骤、容易漏掉的地方和交付标准一起给 AI。

### 第二步：让 AI 先整理流程骨架

先输出目标、范围、角色、前置条件和主流程，确认方向没错后再细化。

### 第三步：补充检查点和异常处理

SOP 的价值不只是步骤，还包括“如何判断做对了”和“出错怎么办”。

### 第四步：让执行者试跑

生成后的 SOP 要让实际执行者按步骤试跑一次，把看不懂、做不到、缺材料的地方补上。

## 使用案例一：内容发布 SOP

### 场景

AI 内容站需要持续发布文章，但每次发布都要检查 Frontmatter、内链、SEO 描述和构建结果。可以用 SOP 固化流程。

### 输入示例

```text
目标：发布一篇新的 Workflow 文章
步骤：选题、写作、检查标题描述、添加内链、运行构建、提交 GitHub
质量标准：页面可访问，SEO 字段完整，无构建错误
常见异常：标签缺失、链接 404、构建失败、标题过长
```

### 预期输出方向

SOP 应包含适用范围、角色分工、前置条件、操作步骤、检查清单和异常处理。对于构建失败和 SEO 字段缺失，应给出处理方式。

## 使用案例二：客户交付前检查 SOP

### 场景

设计或内容交付前，经常出现附件漏发、版本不一致、命名混乱等问题。可以用 Skill 生成交付前检查 SOP。

### 输入示例

```text
目标：客户交付前完成最终检查
步骤：确认交付范围、检查文件命名、核对版本、准备说明文档、发送邮件
质量标准：客户能直接打开、理解和使用交付物
常见异常：附件太大、链接权限未开放、文件版本错误
```

### 使用建议

让 AI 把 SOP 输出成“执行清单 + 审核清单”两部分。执行者按步骤操作，审核者只看关键结果。

## 检查清单

- 每一步是否有明确动作，而不是抽象原则？
- 是否写清楚谁负责、谁确认？
- 是否有前置条件和所需材料？
- 是否定义完成标准？
- 是否覆盖常见异常和处理方式？
- 新人不问人是否也能执行？

## 注意事项

SOP 不是写完就结束。流程变化、工具变化或团队分工变化后，要更新版本记录。对于高风险流程，建议保留人工审核节点，不要完全依赖自动化。
